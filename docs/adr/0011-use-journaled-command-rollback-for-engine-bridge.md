# ADR-0011: Use Journaled Command Rollback For The Engine Bridge

Date: 2026-07-06

Status: Accepted

## Context

The Ardour-derived engine spike proved that a headless C++ session utility can create and open sessions, import audio, create tracks, place regions, save, render, observe a project graph, and replay deterministic fixture command logs.

The next trust problem is rollback. Reson needs user-visible reversibility for agent actions, but AI plans must not directly manipulate low-level Ardour session state or assume that mutable names still identify the same objects.

## Decision

Continue with the Ardour-derived engine path and deepen the C++ session utility bridge as the first command surface.

Each mutation command must be journaled with:

- Command input and schema version.
- Pre-command observation token or graph snapshot.
- Engine-assigned stable IDs touched by the command.
- Post-command observation graph.
- Render or replay verification artifacts when applicable.
- Rollback action type.

For the spike, rollback is implemented as session-level snapshot restore before and after command batches. This is heavier than fine-grained undo, but it is auditable, simple to test, and preserves user trust.

Fine-grained inverse commands may be added later for common operations such as removing an imported region, moving a region back to its previous playlist and position, deleting newly-created tracks, or restoring render/export settings. These inverse commands must still be generated by the bridge from observed state, not by the agent guessing.

Reject direct agent access to Ardour undo stacks, OSC structural edits, and name-only rollback plans as primary rollback mechanisms.

## Consequences

The command bridge needs a durable command journal, snapshot storage policy, observation graph hashing, stale-state rejection, and replay tests.

Session snapshots may be slow or storage-heavy, so the bridge should prune them by batch, risk level, and user confirmation state.

This decision favors correctness and auditability over minimal disk usage during the first engine integration.
